# 픽셀 아트 에디터 - Review 단계 지침

이 문서는 Review 서브에이전트 전용 작업 지침이다. Build 단계는 완료되었고 `apps/pixel-editor/index.html`, `apps/pixel-editor/style.css`, `apps/pixel-editor/editor.js`가 이미 구현되어 있다. 너는 Build를 수행한 에이전트와는 별개의 에이전트로서, 검증자(QA) 입장에서 확인한다.

## 범위

- 검증 대상: `apps/pixel-editor/index.html`, `apps/pixel-editor/style.css`, `apps/pixel-editor/editor.js`.
- 문제를 발견하면 **그 세 파일 안에서만** 직접 수정해도 된다. 그 외 블로그 파일은 건드리지 않는다.
- 검증 결과를 `apps/pixel-editor/review.md`에 작성한다.
- 참고 문서: `apps/pixel-editor/spec.md`(요구사항 원본), `apps/pixel-editor/build-instructions.md`(Build 지침).

## 검증해야 할 것

### 1. 브라우저 동작 확인 (Browser 도구 사용)
- 저장소 루트를 서빙하는 임시 정적 서버(예: `python -m http.server`)를 리뷰 목적으로 띄우고 `apps/pixel-editor/index.html`을 연다(`file://`로 열면 프리뷰 도구가 스크립트를 실행하지 않을 수 있으니 정적 서버 경유를 권장).
- 마우스 클릭으로 도트가 찍히는지, 클릭한 채 드래그(마우스 다운 후 이동)하면 지나간 칸에 연속으로 찍히는지 확인한다.
- 합성 터치 이벤트로 드래그 페인팅이 되는지 확인한다.
- 팔레트에서 다른 색을 클릭하면 현재 색상이 바뀌고 그 색으로 찍히는지 확인한다. 커스텀 컬러피커(`<input type="color">`)로 색을 바꾼 뒤에도 해당 색으로 찍히는지 확인한다.
- 지우개 도구를 선택하고 찍었던 칸을 지우면 실제로 지워지는지(투명하게 되는지) 확인한다.
- "전체 지우기" 버튼이 모든 칸을 비우는지 확인한다.
- 저장 버튼(원본/확대) 클릭 시 실제로 다운로드 요청이 발생하는지 `read_network_requests` 또는 다운로드 이벤트로 확인한다(파일을 실제로 열어 픽셀 내용까지 검증할 필요는 없지만, 최소한 blob URL 생성과 다운로드 트리거가 정상 동작하는지는 확인).
- roving tabindex: Tab으로 그리드에 포커스가 들어간 뒤 방향키로 셀 간 포커스가 이동하는지, Enter/Space로 포커스된 셀에 색이 찍히는지 확인한다(`read_page`로 `tabindex` 값 변화, computer의 key 액션으로 확인).
- 콘솔 에러(`read_console_messages`)가 없는지 확인한다.
- `resize_window`로 모바일 뷰포트(예: 375x812)로 바꿔서 레이아웃이 깨지지 않는지 확인한다.
- 다크/라이트 테마 전환 시(`localStorage`의 `theme` 값을 바꿔 재로드) 정상 반영되는지 확인한다.

### 2. 코드 리뷰
- `pixels` 배열이 유일한 상태이고 캔버스가 그 순수 렌더 결과인지(상태 이중관리 없는지) 확인한다.
- 빈 칸(`null`)에 대해 `fillRect`를 호출하지 않아 PNG가 투명 배경을 유지하는 구조인지 코드로 확인한다.
- 확대 저장 시 `imageSmoothingEnabled = false`가 실제로 설정되어 있는지 확인한다.
- 캔버스(`<canvas>`)에 `aria-hidden="true"`가 있고, 실제 상호작용은 오버레이 버튼이 전담하는 구조인지 확인한다.
- `touch-action: none`이 캔버스/그리드 컨테이너에 적용되어 있는지 확인한다.
- 블로그 다른 파일(css/, js/, templates/, apps/2048/ 등)을 참조하거나 의존하지 않는지(자체 완결) 확인한다.

## 문제 발견 시

- 사소한 버그는 해당 파일에서 직접 수정한다.
- 수정 후에는 반드시 브라우저에서 재확인해서 실제로 고쳐졌는지 검증한다.
- 구조적으로 spec.md와 크게 어긋나는 문제가 있다면 수정하지 말고 review.md에 이슈로 남기고 보고에서 강조한다.
- 검증 과정에서 디버그용 임시 코드를 추가했다면 검증 완료 후 반드시 제거하고, 최종적으로 파일에 남아있지 않은지 확인한다.

## review.md 작성 형식

- 검증 일시, 검증 방법(사용한 도구)
- 항목별 결과 (통과/실패 표시)
- 발견한 문제와 수정 내역 (있다면)
- 최종 결론: 배포(임베드) 가능 여부
